Persistent Python sessions on a pinned interpreter. Every session can `import tui` with no install step: `tui` is the bundled PyO3 PTY driver (spawn and drive PTY-backed processes with full vt100 emulation, scrollback, and optional NumPy access to per-cell data; the API is async-only, so use it from python_exec/python_eval with top-level await). `search` is bundled too: `await search.semantic(query)` runs a read-only semantic search over the shared `index` corpus (code plus Claude/Codex/shell history across the fleet) and returns scored hits, and `await search.grep(pattern)` runs a regex over the same corpus chunks. Neither indexes (the separate `indexer` populates the store), so a query never uploads your local checkout. Scope server-side with `source`/`user`/`repo`/`host`/`project`, e.g. `await search.semantic(q, source=["claude_history"], user=["andrew"])`. Both are also exposed directly as the `search_semantic` and `search_grep` tools. On macOS, `screen` is bundled for native desktop/computer-use automation (screenshots, cursor read/move, synthetic clicks and drags), the same way `tui` covers PTYs and `playwright` covers browsers; see the screen paragraph below. numpy, polars, and matplotlib are preinstalled for data work, as is `asyncssh` for SSH-based remote process control (run commands on remote hosts, SFTP) that fits the shared async event loop, and each session's writable venv keeps `pip install` working for anything else. polars can query Postgres directly: `psycopg` (v3) and SQLAlchemy are bundled, so `pl.read_database(sql, sqlalchemy.create_engine("postgresql+psycopg://user@host/db"))` returns a DataFrame (a raw `psycopg.connect(...)` connection also works); the connectorx/ADBC engines that `read_database_uri` wants are not bundled, so prefer the SQLAlchemy-engine path. `duckdb` is bundled for in-process analytical SQL over CSV/Parquet with no external service — `duckdb.sql(q).pl()` returns a polars DataFrame (and can query a polars frame by name). pyarrow is not bundled (it pulls ~300 MB of arrow-cpp + cloud C++ SDKs the common paths never use); `pip install pyarrow` in-session if you need explicit Arrow tables (`pl.to_arrow()`, `duckdb.sql(q).fetch_arrow_table()`). `httpx` is bundled for HTTP on the shared async loop: `httpx.get(url)` for one-offs and `async with httpx.AsyncClient() as c: await c.get(url)` for the async path. The official Google client is bundled too (`googleapiclient` from google-api-python-client, with `google_auth_oauthlib` and `google.oauth2` for the OAuth flow), so a session can drive the Gmail and Calendar APIs directly: `build("gmail", "v1", credentials=creds)`. No credentials or tokens are bundled; bring your own from rbw/op. `playwright` is preinstalled too, with its browsers already wired (PLAYWRIGHT_BROWSERS_PATH is set), so `from playwright.async_api import async_playwright` drives a real browser with no `playwright install` step; use the async API so it shares the session event loop. Reuse a browser instead of relaunching one per call: a fresh `chromium.launch()` every turn wastes startup time and throws away cookies and logins, so launch once and reuse the handle (see the playwright paragraph below), and connect over CDP to share one browser across separate sessions. Chromium and Firefox are the reliable engines; WebKit is known to be flaky under Nix. python_eval and python_exec return rendered images automatically: any open matplotlib figure, PIL image, `display()`ed object, or value with a `_repr_png_` comes back as an inline image alongside the text output.

Intent: python_eval and python_exec require a one-line `intent` describing what the call is doing, phrased in the present continuous ("counting rows in the orders parquet", not "count rows" or "run code"). It titles the run's card on the live dashboard, so the board reads as a list of plain-language actions rather than raw code; keep it short and specific.

Execution model: each session is single-threaded with one persistent asyncio event loop, reused across calls. Write top-level `await` directly (e.g. `rows = await pool.fetch(sql)`); because the loop is shared, an async client or pool created in one call stays usable in later ones. Do not call `asyncio.run()`: inside an awaited block it raises `RuntimeError: asyncio.run() cannot be called from a running event loop`, and outside one it spins up a throwaway loop your resources cannot outlive. `asyncio.get_running_loop()` resolves only inside awaited code. A blocking synchronous call (heavy CPU, `time.sleep`, sync network I/O) freezes the whole session, so reach for the async equivalent.

Timeouts and cancellation: python_eval and python_exec abandon a call that runs longer than 60 seconds (the search tools allow 600s) and return a timeout error. Raise the budget for legitimately long work by passing `timeout_secs` (e.g. a long training run or download); there is no infinite setting. A call you cancel from the client (MCP `notifications/cancelled`) is abandoned the same way, within a fraction of a second. Either way the session's worker is restarted, so a runaway cell never wedges the session — but the restart is a fresh interpreter, so in-memory state (imported modules, variables, open clients) from before the abandoned call is lost. Different sessions are independent: a hung or long call in one session does not block calls to another (the one exception is the few seconds a brand-new session spends building its venv, which briefly serializes other sessions' first call).

Driving a terminal with `tui`: spawn and control a process in one block, awaiting every I/O step.

    from tui import Tui, Key
    async with Tui("python", "-q") as t:
        await t.enter("print(2 + 2)")
        snap = await t.wait_for("4", timeout=2.0)
        print(snap.text)

Every method that touches the terminal is a coroutine you must `await`: `send`, `enter`, `read`, `viewport`, `text`, `snapshot`, `wait_for`, `resize`, `kill`, `close`. Only the cached accessors are synchronous: `id`, `command`, `args`, `size`, `is_alive`, `exit_code`. Send keystrokes with the `Key` enum (`Key.ENTER`, `Key.ctrl("c")`, arrows). The `async with` block calls `close()` on exit, so an editor or REPL that ignores Ctrl+C still goes away.

Terminals auto-show in the dashboard. The first `Tui(...)` publishes this process, so a running `nix run .#dashboard` (it watches `tui.socket_dir()`) renders every spawned terminal with no `tui.publish()` call. Set `IX_TUI_AUTOPUBLISH=0` to opt out. The dashboard is a generic canvas of panes: a producer can also publish HTML or structured-data panes, not just terminals (`nix run .#dashboard demo` shows one of each).

Driving a browser with `playwright`, reuse-first. Within one session, launch once and keep the handle in a module global; the persistent loop keeps the browser and its cookies alive across calls, so later calls reuse it instead of paying a fresh launch.

    from playwright.async_api import async_playwright

    if "browser" not in globals():
        p = await async_playwright().start()
        browser = await p.chromium.launch()  # one launch, reused below
    page = await browser.new_page()
    await page.goto("https://example.com")
    print(await page.title())

To share one browser across separate sessions or processes, connect over CDP (Chromium only). Decide connect-vs-launch on a recorded endpoint, not a swallowed exception: the first session launches a persistent browser on a fixed debug port and records the endpoint; later sessions read the file and attach. Connecting reuses the running browser's open contexts and their state; `connect_over_cdp` returns it via `browser.contexts[0]`.

    import json
    from pathlib import Path
    from playwright.async_api import async_playwright

    endpoint = Path("/tmp/ix-mcp-chromium-cdp")
    p = await async_playwright().start()
    if endpoint.exists():
        cdp = endpoint.read_text()
        print(f"reusing browser at {cdp}")
        browser = await p.chromium.connect_over_cdp(cdp)
        context = browser.contexts[0]
    else:
        port = 9222
        cdp = f"http://localhost:{port}"
        print(f"no recorded endpoint, launching chromium at {cdp}")
        await p.chromium.launch(args=[f"--remote-debugging-port={port}"])
        endpoint.write_text(cdp)
        browser = await p.chromium.connect_over_cdp(cdp)
        context = browser.contexts[0]
    page = await context.new_page()
    await page.goto("https://example.com")

`connect_over_cdp` detaching with `browser.close()` leaves the launched browser running for the next attach; the launching session owns the real shutdown. If the recorded endpoint is stale (the browser exited), the connect raises rather than silently relaunching: delete the file and run the launch branch again.

Driving the native desktop with `screen` (macOS only). `import screen` for synchronous CoreGraphics helpers: `screen.capture()` returns the full desktop as a `PIL.Image` (pass a `screen.Rect(x, y, w, h)` for a region; the worker renders a returned PIL image inline, so just end a cell with the image), `screen.capture_ndarray()` returns an `(H, W, 3)` uint8 RGB array for pixel math and diffing, `screen.cursor()` returns the pointer as a `screen.Point`, and `screen.move(x, y)` warps it. Capture and `move` need no special permission. Synthetic input does: `screen.click(x, y)`, `screen.drag(x1, y1, x2, y2)`, and `screen.mouse_down`/`screen.mouse_up` require the host process to be trusted for macOS Accessibility (System Settings > Privacy & Security > Accessibility). An untrusted process has its posted events silently discarded by the OS, so a click looks like it worked while nothing happened. To avoid that silent no-op, these helpers check `screen.accessibility_trusted()` first and raise `screen.AccessibilityNotTrusted` (naming the untrusted process) instead of posting a dropped event. Call `screen.accessibility_trusted()` before relying on synthetic input, and treat a False result as "grant Accessibility to the host running ix-mcp and restart it," not as a transient failure. Recent macOS also gates `capture()` behind Screen Recording permission; an empty/None capture raises with that hint.

    import screen
    shot = screen.capture()                 # full desktop, rendered inline
    if screen.accessibility_trusted():
        screen.click(100, 200)              # synthetic input, permission-checked
    else:
        print("grant Accessibility to the ix-mcp host, then restart it")
