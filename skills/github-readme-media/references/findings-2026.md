# GitHub README media rendering: 2026 reference

Every "observed" claim here was verified live in a real Chromium 147 browser (the
engine github.com serves) by loading actual github.com READMEs and the GitHub
markdown API, not from docs alone. Where docs and observation conflict,
observation wins.

## TL;DR

For a programmatically-regenerated, autoplaying, looping, dark-mode-aware
terminal demo: commit an **animated WebP** (or AVIF) and reference it with a
`<picture>` element using `prefers-color-scheme`. This is the only approach that
is simultaneously (a) committed / CI-generatable with no manual web upload, (b)
autoplaying + looping, and (c) dark/light aware. Inline `<video>` cannot do
autoplay/loop and cannot be generated in CI.

## 1. Inline video in README markdown

**Author-written `<video>` tags are stripped by the markdown sanitizer.**
Observed via `POST https://api.github.com/markdown`: a
`<video src=... autoplay loop muted playsinline controls poster width>` and a
`<video><source></video>` both came back with the `<video>` element removed
(only a stray orphan `<source>` survived). You cannot hand-author a working
`<video>` in a README.

**The `<video>` you see in some READMEs is generated by GitHub, not authored.**
When you paste a bare attachment URL on its own line, GitHub's repo-context
renderer replaces it with a player it builds itself. Observed on
`github.com/george-hawkins/video-in-github-markdown`: the rendered player was
`<video src="https://private-user-images.githubusercontent.com/.../...mp4?jwt=..." muted>`
with `controls=true`, `autoplay=false`, `loop=false`, and the author's
`width`/`autoplay`/`loop`/`<source>` all stripped.

- **Autoplay/loop: not possible.** GitHub forces `controls` on and ignores
  `autoplay`/`loop`. The maintainer who tested every permutation states: "There
  appears to be no way to enable autoplay or looping for such videos."
- **Allowed attributes:** effectively only `muted` survives; `controls` is
  force-added; everything else is dropped.

**Committed file vs uploaded attachment:**

- A committed video (relative path or `raw.githubusercontent.com` / `/raw/` URL)
  does **not** become an inline player. Observed: `![](output.mp4)`, a bare
  `/raw/master/output.mp4` URL, and `<video src="output.mp4">` none rendered as
  video. Only the two drag-and-drop **user-attachments** URLs produced a working
  `<video>`.
- An uploaded attachment **does** become a player, but the bare URL must be left
  naked (no markdown link, no backticks, no `<video>` wrapper) for the
  auto-detection to fire.

**The drag-and-drop attachment flow:**

- Dragging a video into an issue/PR/README editor uploads to GitHub's CDN and
  inserts `https://github.com/<owner>/<repo>/assets/<user-id>/<guid>` (newer
  canonical: `https://github.com/user-attachments/assets/<guid>`). On render it
  resolves to `https://private-user-images.githubusercontent.com/<id>/<guid>.mp4?jwt=<token>`
  for private repos (observed live).
- The asset is **not in your repo**; cloning does not fetch it. The committed
  README just holds the magic URL.
- It **cannot be generated programmatically / in CI.** There is no official REST
  or GraphQL API for uploading these attachments; the upload requires a
  logged-in browser session cookie (`POST https://github.com/upload/policies/assets`
  returns 422 with a PAT). Unofficial workarounds drive a browser cookie (e.g.
  the `gh-image` CLI extension), but there is no first-party programmatic path.
  This disqualifies inline video for a "no manual drag-drop" requirement.

**Size limits (official):**

- Images & GIFs: 10 MB
- Videos: 10 MB (free) / 100 MB (paid)
- Other attached files: 25 MB
- Supported upload video formats: `.mp4`, `.mov`, `.webm`; H.264 recommended.
- Files committed to git: warning at 50 MB, hard block over 100 MB per file
  (push rejected), then Git LFS. These git limits are separate from attachment
  limits.

## 2. Animated image formats (the viable autoplay path)

All four tested for animation in Chromium 147; the first three additionally
confirmed rendering AND animating live on github.com in a real README.

| Format | GitHub renders | Autoplays + loops | Notes |
|---|---|---|---|
| Animated GIF | yes (live) | yes, always | universal, huge files, 256-color cap, bands gradients/text |
| Animated WebP | yes (live, `![]()` and `<img>`) | yes (loops by default in the player; encode `-loop 0`) | ~3.5x smaller than GIF at similar quality, 24-bit |
| Animated AVIF | yes (live, committed `/raw/`) | yes | smallest + best quality; not in GitHub's official list |
| APNG | animates in Chromium | yes | 24-bit + alpha; larger than WebP; no advantage |

Observed on `github.com/george-hawkins/video-in-github-markdown`: animated WebP
(`/raw/master/output-4.webp`), AVIF (`/raw/master/output-30.avif`), and GIF all
rendered with `naturalWidth=1280` and all three animated (screenshots 900ms
apart differed). All served from committed repo files via `/raw/`, no
user-attachments.

Concrete sizes (same 720p 24fps clip, maintainer's measurements): GIF 8.2 MB,
WebP 2.3 MB, AVIF 557 KB. The same clip as CRF-28 MP4 was ~616 KB, but MP4 cannot
autoplay inline. The maintainer noted WebP showed slight glitching before the
loop that AVIF did not.

**AVIF caveat:** absent from GitHub's official supported-formats list
(`.png .gif .jpg/.jpeg .svg` + video). It works today when committed and served
via `/raw/` (raw.githubusercontent.com returns `Content-Type: image/avif`,
observed 200) and modern browsers decode it, but it is undocumented (more "could
change" risk) and cannot be uploaded as a drag-drop attachment.

Ranking for a 10-30s terminal demo:
- Quality: AVIF ≈ WebP ≫ APNG > GIF (GIF's 256-color cap wrecks anti-aliased text).
- Size: AVIF < WebP < APNG < GIF (GIF ~15x AVIF).
- Autoplay/loop reliability: GIF ≈ WebP > AVIF ≈ APNG.

For terminal demos, **WebP is the sweet spot**: 24-bit keeps mono text crisp,
~3.5x smaller than GIF, reliable autoplay+loop, broadly supported. AVIF for
minimum bytes if you accept undocumented status.

## 3. Dark/light mode media swapping

Use `<picture>` + `prefers-color-scheme`. Works with animated images.

Observed live on `github.com/kera-cudmore/example-light-dark-github-readme-images`:
emulating `prefers-color-scheme: dark` selected the dark image and `light` the
light one (`img.currentSrc`). Confirmed via the markdown API that
`<picture><source media="(prefers-color-scheme: dark)" srcset=...>` survives
sanitization (wrapped in a `themed-picture` web component, `media`+`srcset`
preserved).

- Works with `<img>` sources: yes (standard form).
- Works with animated WebP/GIF/AVIF: yes; animation and theme-selection are
  independent.
- Works with `<video>`: no, author `<video>` is stripped.
- The older `#gh-dark-mode-only` / `#gh-light-mode-only` fragment method still
  functions but is superseded by `<picture>`.

```html
<picture>
  <source media="(prefers-color-scheme: dark)"  srcset="docs/demo-dark.webp">
  <source media="(prefers-color-scheme: light)" srcset="docs/demo-light.webp">
  <img alt="demo" src="docs/demo-light.webp" width="800">
</picture>
```

## 4. Recommendation (flat, minimal, monospace terminal demo, ~15-30s)

1. **Animated WebP committed to the repo, wrapped in `<picture>` for dark/light.**
   Best overall: autoplays, loops, crisp 24-bit text, ~3.5x smaller than GIF,
   CI-generatable, dark-mode aware, documented + verified live.
2. **Animated AVIF, same wrapper.** ~4x smaller and cleaner than WebP, verified
   live, but undocumented and not a valid drag-drop type. Choose if byte size is
   paramount.
3. **Animated GIF** as a universal fallback only. Reliable but big and banded.
4. **Inline `<video>` (user-attachments)** rejected: no autoplay, no loop, forced
   controls, no programmatic/CI upload path.

Generation (no manual web steps): render the terminal session, then encode with
ffmpeg. WebP: `ffmpeg -i demo.mp4 -loop 0 -vf "scale=-1:720" demo.webp` (the
`-loop 0` is required). AVIF: `ffmpeg -i demo.mp4 -vcodec libaom-av1 -vf
scale=-1:720 -crf 30 demo.avif` (loops by default). Keep each file under 10 MB.
Render two theme variants and commit both.
